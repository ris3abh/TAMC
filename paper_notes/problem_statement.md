# Problem Statement

## 1. Setting

Let `x_1, x_2, ..., x_t, ...` be a univariate stream observed sequentially.
The stream is generated by an underlying dynamical system whose governing
parameters may change at unknown, unlabeled times. We call each interval
during which the governing dynamics stay fixed a **regime**.

A **frozen forecaster**

```text
f_theta: R^L -> R^H
y_hat_t = f_theta(x_{t-L:t})
```

maps a context window of length `L` to a forecast of horizon `H`. Critically,
`theta` is fixed at deployment time and is **never updated** — no gradient
step, no fine-tuning, no weight change of any kind occurs after `f_theta`
is fit. `f_theta` is fit once, on data drawn only from the **source
regime** (defined below), before any shift occurs.

## 2. Strict Causal Information Constraint

At every time `t`, any computation — the forecaster's prediction, the
topological drift score, the adapter's gate or correction — may use only
information that would actually be available to an online system observing
the stream causally:

```text
available at time t:  x_1, x_2, ..., x_t
unavailable at time t: x_{t+1}, x_{t+2}, ...   (no future labels)
                        any ground-truth y for forecasts already made
```

This rules out two common sources of leakage in time-series TTA
evaluation:

1. **Future-label leakage**: using `x_{t+1:t+H}` (the actual future values
   being forecast) to compute or calibrate anything at time `t`.
2. **Whole-series statistics leakage**: computing a normalization,
   threshold, or source prototype from data that spans both regimes,
   rather than only the source regime or only the causal past.

Every experiment in this repo enforces this by construction: source
prototypes and frozen-forecaster fits are built only from the source
segment (`series[:shift_index]`), and all rolling/sliding-window
computations index strictly backward from the current step
(`series[t-window:t]`, never `series[t:t+window]`).

## 3. Source Regime, Shifted Regime, Context Length, Forecast Horizon

- **Source regime**: the initial segment of the stream,
  `x_1, ..., x_{shift_index}`, generated under one fixed set of dynamical
  parameters. `f_theta` is fit only on this segment. A single reference
  window from this segment also defines the topological source prototype
  `D_source` (see [methodology.md](methodology.md#1-topological-drift-signal)).
- **Shifted regime**: the segment after the shift,
  `x_{shift_index+1}, ...`, generated under a different fixed (or
  qualitatively different, e.g. chaotic vs. stable) set of dynamical
  parameters. No data from this segment is used to fit `f_theta` or to
  build `D_source`.
- **Context length** (`L` / `context_length`): the number of most-recent
  observations fed to a forecaster at prediction time,
  `x_{t-L}, ..., x_{t-1}`.
- **Forecast horizon** (`H` / `horizon`): the number of future steps a
  forecaster predicts in one call, `y_hat_t = [\hat{x}_t, ..., \hat{x}_{t+H-1}]`.

In the current experiments, `shift_index` marks a single, controlled
regime boundary (e.g. `n_per_regime=700` samples per regime); the
underlying generators (`generate_sine_to_quasiperiodic`,
`generate_logistic_map_regime_shift`, `generate_lorenz_regime_shift`) are
known exactly, since this is a controlled synthetic setting used to
validate the method before any real-data evaluation.

## 4. What "Forward-Only" Means Here

"Forward-only" has two distinct senses in this project, and both apply
simultaneously:

1. **No weight updates** (the test-time-adaptation sense): `f_theta`'s
   parameters never change after fitting. Any adaptation happens in
   output space (a gate, a blend, a residual correction) or by running a
   second, separately-defined forecaster — never by backpropagating into
   `f_theta`.
2. **No gradients through the adapter's decision** (the
   zeroth-order/forward-only-optimization sense, relevant when an
   adapter's own small parameter vector is being searched, e.g.
   `ForwardOnlyOptimizer` in `src/adapters.py`): any such search uses only
   forward evaluations of a fitness function, never backpropagation.

Both senses share the same motivating constraint: TAMC should work for any
frozen forecaster treated as a black box, including ones too large,
too expensive, or too inaccessible to fine-tune at test time.

## 5. What TAMC Adds On Top of This Setup

Given the constraints above, the open question is *how* to decide when and
how much output-space adaptation should occur, using only the causal past
and no labels. TAMC's proposal is to compute a topological drift score
`delta_t` from the delay-reconstructed attractor of the causal past, and
use it strictly as a **meta-control** signal (a multiplier or selector)
over a forward-only adapter — never as an additive term in an optimization
objective and never as a substitute for the strict causal/no-future-label
constraints above. See
[methodology.md](methodology.md#3-tamc-as-meta-control-not-an-additive-loss-term)
for the full mechanism.
